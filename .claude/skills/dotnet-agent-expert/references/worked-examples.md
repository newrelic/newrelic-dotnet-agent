## Example 1: Where does `queueDuration` get sent?
**Question (as asked):** "Where does the `queueDuration` attribute get set, and which payload does it actually ship in?"
**Detected persona:** Software engineer — names a specific attribute and asks "where … set" and "which payload" (internals).
**Trace:** `HttpContextActions.StoreQueueTime` (`src/Agent/NewRelic/Agent/Extensions/Providers/Wrapper/AspNet/Shared/HttpContextActions.cs:69-81`) → `Transaction.SetQueueTime` (`src/Agent/NewRelic/Agent/Core/Transactions/Transaction.cs:966-968`) → `TransactionMetadata.SetQueueTime`/`QueueTime` (`src/Agent/NewRelic/Agent/Core/Transactions/TransactionMetadata.cs:141-143, :245-247`) → `AttributeDefinitionService.QueueDuration` (`src/Agent/NewRelic/Agent/Core/Attributes/AttributeDefinitionService.cs:397-403`) → `TransactionAttributeMaker` (`src/Agent/NewRelic/Agent/Core/Transformers/TransactionTransformer/TransactionAttributeMaker.cs:66`) → `TransactionEventMaker.GetTransactionEvent` / `ErrorEventMaker` (`src/Agent/NewRelic/Agent/Core/Transformers/TransactionTransformer/TransactionEventMaker.cs:25-33`; `ErrorEventMaker` is the parallel error-event path).
**Answer:**
> `queueDuration` originates in the ASP.NET wrapper: `HttpContextActions.StoreQueueTime` (`HttpContextActions.cs:69-81`) computes `now - workerRequestStartTime` and calls `SetQueueTime`, which stores it on `TransactionMetadata.QueueTime` (`Transaction.cs:966-968` → `TransactionMetadata.cs:141-143, :245-247`). The attribute itself is defined in `AttributeDefinitionService.QueueDuration` (`AttributeDefinitionService.cs:397-403`) as an **intrinsic** named `queueDuration` (value in seconds) that `.AppliesTo` exactly two destinations: `TransactionEvent` and `ErrorEvent`. `TransactionAttributeMaker` populates it (`TransactionAttributeMaker.cs:66`) and `TransactionEventMaker`/`ErrorEventMaker` carry it into the wire models. So it ships only on the transaction event and the error event — not on transaction traces or span events. (The same queue time separately drives the unscoped metric `WebFrontend/QueueTime` via `TransactionTransformer`, but that's a metric, not the attribute.)

## Example 2: US vs EU collector selection
**Question (as asked):** "How does the agent decide whether to connect to the US vs EU collector? A customer thinks their EU data is going to the US."
**Detected persona:** Technical support — "a customer thinks…" / diagnosing a wrong-region symptom (behavior + how to verify, with code as evidence).
**Trace:** `DefaultConfiguration.CollectorHost` (`src/Agent/NewRelic/Agent/Core/Configuration/DefaultConfiguration.cs:754`) → `ConnectionInfo` ctor (`src/Agent/NewRelic/Agent/Core/DataTransport/ConnectionInfo.cs:45-47`) → `ConnectionInfo.GetCollectorHost` (`src/Agent/NewRelic/Agent/Core/DataTransport/ConnectionInfo.cs:60-86`).
**Answer:**
> The agent auto-selects the collector from the license key, so EU vs US is usually not a separate setting. In `ConnectionInfo.GetCollectorHost`, an explicit `host` config value (`service.host` / `NEW_RELIC_HOST`) wins first (`ConnectionInfo.cs:67-70`); otherwise the license key is matched against the region regex `^.+?x` (`ConnectionInfo.cs:74`) and the prefix (minus the trailing `x`) is spliced into `collector.nr-data.net` — e.g. an `eu01x…` key resolves to `collector.eu01.nr-data.net` (`ConnectionInfo.cs:78-81`). A plain (non-region) key falls back to the US default `collector.newrelic.com` (`ConnectionInfo.cs:85`). To verify: confirm the customer's license key carries the `eu01` region prefix; if it does, data routes to EU automatically. If they need to force a host, set `service.host` explicitly — it overrides the key-based selection. (Also note a collector-supplied redirect host takes over for subsequent requests.) Backing config resolution: `DefaultConfiguration.CollectorHost` (`DefaultConfiguration.cs:754`).

## Example 3: RabbitMQ attributes
**Question (as asked):** "What attributes does the agent collect for RabbitMQ, and where are they attached in the code?"
**Detected persona:** Software engineer — asks for the attribute set *and* where they're attached in code (internals).
**Trace:** `RabbitMqHelper.CreateSegmentForPublishWrappers` (`src/Agent/NewRelic/Agent/Extensions/Providers/Wrapper/RabbitMq/RabbitMqHelper.cs:62-110`) / `HandleBasicDeliverWrapper.BeforeWrappedMethod` (`src/Agent/NewRelic/Agent/Extensions/Providers/Wrapper/RabbitMq/HandleBasicDeliverWrapper.cs:40-70`) → `MessageBrokerSegmentData` (`src/Agent/NewRelic/Agent/Core/Segments/MessageBrokerSegmentData.cs:18-56`) → `MessageBrokerSegmentData.SetSpanTypeSpecificAttributes` (`src/Agent/NewRelic/Agent/Core/Segments/MessageBrokerSegmentData.cs:110-138`) → `AttributeDefinitionService` (`src/Agent/NewRelic/Agent/Core/Attributes/AttributeDefinitionService.cs:1194-1245`).
**Answer:**
> The publish wrappers (`RabbitMqHelper.cs:62-110`) and the consume wrapper (`HandleBasicDeliverWrapper.cs:40-70`) call `StartMessageBrokerSegment`, passing the destination (from the routing key), the RabbitMQ vendor, the routing key, and the server address/port resolved by reflection from `Session.Connection.Endpoint`. Those land on `MessageBrokerSegmentData` (`MessageBrokerSegmentData.cs:18-56`), and `SetSpanTypeSpecificAttributes` (`MessageBrokerSegmentData.cs:110-138`) emits the span/trace attributes via `AttributeDefinitionService` (`AttributeDefinitionService.cs:1194-1245`): `server.address`, `server.port`, `messaging.system`, `messaging.destination.name`, `message.routingKey`, `messaging.rabbitmq.destination.routing_key`, and on the consume side `message.queueName` and `messaging.destination_publish.name`; plus the `span.kind` intrinsic (producer/consumer). It also propagates distributed-trace context through the message `Headers`. Server address/port are best-effort — if the reflection fails they're omitted and logged at Warn.

## Example 4: Does the agent support MongoDB.Driver 3.7?
**Question (as asked):** "Does the agent support MongoDB 3.7? A prospect wants to confirm before they commit."
**Detected persona:** Sales / solution — "does it support…" with explicit pre-sales framing ("a prospect wants to confirm"). Capability answer, no citations.
**Trace:** Capability determination (no line citations): read the `<match>` version bounds in the `MongoDb26` `Instrumentation.xml` (modern 2.x/3.x driver) versus the legacy `MongoDb` `Instrumentation.xml` (1.x driver), and the supported range pinned in the integration test package files. Most modern matches carry no version bounds; the only gated ones switch type/assembly at the driver's 3.0 packaging boundary, with the `minVersion="3.0.0"` branches applying at 3.7.
**Answer:**
> Yes — the agent supports MongoDB.Driver 3.7. (This refers to the MongoDB.Driver NuGet package, not the MongoDB server version.) MongoDB.Driver instrumentation covers roughly version 2.3 through the latest tested 3.x — verified up to 3.9 — with no upper version cap, so 3.7 falls comfortably inside the supported range. The agent automatically selects the correct instrumentation at the driver's 3.0 packaging change (when `MongoDB.Driver.Core` merged into `MongoDB.Driver`), and support for MongoDB.Driver 3.x and above is an explicitly released feature. A separate, older instrumentation path also covers the legacy 1.x driver API. (I can point you to the exact source and changelog entry if useful.)
